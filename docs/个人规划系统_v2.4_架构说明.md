# 个人规划系统 v2.4 架构说明

> 本文件用于交给 Codex 阅读，作为后续开发 Skills、自定义 Obsidian 插件、任务数据层和鱼骨主视图的架构基线。  
> 当前版本：v2.4  
> 状态：结构层级与实现取向已确认，可进入 Skills 初稿与自定义插件原型设计阶段。  

---

## 0. 总体定位

本系统是一个基于 **Obsidian + Codex + 自定义插件 + Skills** 的个人主线规划系统。

核心目标不是普通待办清单，而是构建一个：

```text
原始输入 → 任务标准化 → 标准任务 md → 鱼骨主视图 → 辅助模块联动
```

的可视化个人规划系统。

最终首页核心 UI 是：

```text
横向日期轴 + 自定义数量主体主线 + 鱼骨任务分支 + relation 关系线 + 辅助模块
```

其中：

- Codex 是系统操控中枢。
- Obsidian 是运行与显示平台。
- Obsidian Vault 是数据存储区。
- 自定义 Obsidian 插件负责核心 UI、任务读写、任务索引、鱼骨视图。
- Codex Skills 负责任务理解、任务标准化、每日总结、插件开发规范等可复用流程。
- 现成 Obsidian 插件优先用于辅助模块，可通过 CSS 或轻度魔改适配 UI。

# 0. 环境与执行层

## 0.1 定位

环境与执行层用于明确：谁负责控制、谁负责运行、谁负责存储、谁负责展示。

当前确认：

```text
Codex = 系统操控中枢
Obsidian = 运行与显示平台
Obsidian Vault = 数据存储区
自定义 Obsidian 插件 = 核心执行器与主视图
Codex Skill = 规则与理解层
现成插件 = 辅助模块优先复用
Git / Backup = 安全保护机制
```

## 0.2 Codex 的职责

用户会给 Codex 较高访问权限，用于自动化管理 Obsidian 和开发项目。

Codex 需要负责：

```text
- 管理 Obsidian Vault 文件
- 写入和修改标准任务 md
- 维护自定义插件代码
- 维护 Skills
- 调用 Skills 理解输入内容
- 生成每日总结
- 更新 task-index
- 调整配置文件
- 执行构建、测试、校验
- 在适当检查点向用户汇报结果
```

## 0.3 Obsidian 的职责

Obsidian 负责：

```text
- 打开和运行用户 Vault
- 运行自定义插件
- 显示鱼骨主视图
- 显示标准任务 md 文件
- 显示辅助模块
- 承载用户日常记录、修改、查看
```

## 0.4 自定义插件的职责

自定义插件负责：

```text
- 提供主视图 Fishbone Timeline View
- 读取标准任务 md
- 读取 mainline 配置
- 读取 task-index
- 显示任务节点、主线、日期轴和 relation 关系线
- 支持任务勾选、打开、编辑、状态更新
- 提供快速输入入口
- 提供候选任务预览
- 执行任务 md 写入
- 更新任务索引
```

## 0.5 Codex Skill 的职责

Skills 不直接替代插件，而是给 Codex 提供长期稳定的规则与流程。

建议至少包含：

```text
planning-task-skill
obsidian-plugin-dev-skill
obsidian-vault-ops-skill
daily-summary-skill
```

# 1. 输入与任务标准化层

## 1.1 层级定位

输入层和任务标准化层已经合并，统一称为：

```text
输入与任务标准化层
```

合并原因：

```text
已有文件读取、自然语言输入、任务理解、任务提取、任务标准化、统一任务 md 生成，均主要由 Codex + Skill 协同完成。
```

## 1.2 输入来源

输入来源包括：

```text
1. Codex 自然语言输入
2. Obsidian 快速输入
3. 手动任务输入
4. 已有文件扫描
```

已有文件包括：

```text
- 日记
- 项目笔记
- 学习笔记
- 论文笔记
- 临时想法
- 会议记录
- 实验记录
- 复盘记录
```

## 1.3 主要功能

本层需要完成：

```text
读取内容
↓
理解内容
↓
提取任务
↓
判断日期
↓
判断所属主线
↓
判断优先级
↓
判断 relation 关系
↓
生成统一任务 md
↓
写入标准任务数据层
```

## 1.4 输入输出关系

示例输入：

```text
明天上午先修训练脚本 bug，修完后再跑对比实验，实验结果后面论文要用。
```

应生成候选任务：

```text
任务 1：修复训练脚本 bug
主线：项目
日期：明天
优先级：高

任务 2：跑对比实验
主线：项目
日期：明天
优先级：中
relation：依赖“修复训练脚本 bug”

任务 3：整理实验结果用于论文
主线：论文
日期：明天或待确认
relation：受“跑对比实验”支撑
```

## 1.5 这一层的实现判别

```text
主要实现：Codex + planning-task-skill + 自定义插件
辅助实现：QuickAdd / Templater / Daily Notes 可选
```

现成插件只作为辅助输入入口，不作为核心任务理解引擎。

# 2. 标准任务数据层

## 2.1 层级定位

标准任务数据层是整个系统的唯一可信数据源。

主视图、今日聚焦、本周重点、进度概览、状态窗口、每日总结都应从这一层读取数据。

## 2.2 核心原则

```text
- 标准任务数据由我们自己定义。
- 现成插件只能读取或展示标准任务数据。
- 不能让现成插件的数据格式反过来限制系统。
- 所有任务关系统一使用 relations 字段。
- 主线数量、名称、颜色、排序均由用户自定义。
```

## 2.3 推荐目录结构

```text
PlanningSystem/
├─ Tasks/
│  └─ 2026/
│     └─ 06/
│        ├─ 2026-06-28_项目_修复训练脚本bug.md
│        └─ 2026-06-28_论文_整理实验结果.md
│
├─ Mainlines/
│  └─ mainlines.json
│
├─ Index/
│  └─ task-index.json
│
├─ DailyReports/
│  └─ 2026-06-28_每日总结.md
│
├─ Inbox/
│  ├─ quick-notes.md
│  └─ unprocessed-inputs.md
│
└─ Config/
   └─ planner-config.json
```

## 2.4 统一任务 md 文件建议结构

每个任务建议一个独立 Markdown 文件。

示例：

```markdown

type: planning-task
task_id: task_20260628_001
title: 修复训练脚本 bug
```

## 2.6 relations 字段设计

任务关系统一使用 `relations` 字段。

不再单独拆分：

```text
relates
depends_on
blocks
```

统一为：

```yaml
relations:
  - target: "[[目标任务]]"
    type: "前置 | 依赖 | 阻塞 | 参考 | 支撑 | 影响 | 普通关联"
    direction: "in | out | both"
    label: "显示在线上的短标签"
    note: "详细说明"
```

建议关系类型：

```text
普通关联
前置
依赖
阻塞
参考
支撑
影响
```

鱼骨图显示规则：

```text
普通关联：细虚线
前置 / 依赖：带箭头虚线
阻塞：醒目颜色虚线
参考：弱化虚线
支撑 / 影响：带标签虚线
```

## 2.7 任务状态

建议状态：

```text
todo        待办
doing       进行中
done        已完成
blocked     阻塞
canceled    取消
inbox       未确认 / 收集箱
```

辅助模块重点显示：

```text
待办
进行中
已完成
```

辅助插件用途：


# 3. 主视图层

## 3.1 层级定位

主视图层是系统核心 UI。

它必须由自定义 Obsidian 插件实现。

现成插件只能作为交互参考，不能替代主视图。

## 3.2 主视图核心形态

```text
横向日期轴
+
自定义数量主体鱼骨主线
+
每日任务鱼骨分支
+
relation 虚线关系
+
可交互任务节点
+
可拖动 / 滚动 / 缩放画布
```

## 3.3 横向日期轴

要求：

```text
- 日期节点横向均匀分布
- 没有任务的日期可以压缩或隐藏
- 有任务日期优先显示
- 支持跳转到今天
- 支持跳转指定日期
```

可存在两种模式：

```text
连续日期模式：每一天都显示
压缩日期模式：只显示有任务的日期或关键日期
```

## 3.4 自定义主体鱼骨主线

要求：

```text
- 主线数量由用户自定义
- 主线名称由用户自定义
- 主线颜色由用户自定义
- 主线排序由用户自定义
- 每条主线显示为一根粗壮脊骨
- 主线可折叠、隐藏、分组、筛选、固定
```

默认主线可以是：

```text
项目
学习
论文
生活/习惯
```

但不能写死。

## 3.5 每日任务鱼骨分支

每个任务挂载到：

```text
某一天 + 某条主线
```

视觉表现：

```text
任务节点分布在主线脊骨上下两侧，形成鱼骨分支。
```

任务节点应显示：

```text
- 任务标题
- 状态
- 优先级
- 是否完成
```

## 3.6 任务完成交互

最小交互：

```text
- 点击任务节点：打开任务 md 文件
- 勾选任务节点：标记完成
- 取消勾选：恢复未完成
```

后续扩展：

```text
- 右键编辑任务
- 拖动调整日期
- 拖动调整主线
- 修改优先级
- 修改 relation
```

## 3.7 任务关系虚线

要求：

```text
- 关系线为虚线
- 线上有箭头
- 线上显示关系标签
- 支持跨日期连接
- 支持跨主线连接
- 根据关系类型改变样式
```

示例：

```text
修复 bug ──前置──▶ 跑对比实验
跑对比实验 ──支撑──▶ 论文结果分析
阅读 Transformer ──参考──▶ 方法设计
作息管理 ──影响──▶ 学习效率
```

## 3.8 横向拖动 / 滚动 / 双重缩放查看

此处的“缩放”包含两种：

```text
1. 时间尺度缩放
2. 画布整体缩放
```

### 时间尺度缩放

控制：

```text
- 日期轴显示密度
- 任务节点聚合程度
- 当前可见时间范围
```

示例：

```text
日视图：显示每一天和具体任务
周视图：按周显示任务聚合
月视图：显示一个月主线推进
总览视图：只显示关键任务 / 里程碑
```

### 画布整体缩放

控制：

```text
整个鱼骨画布的视觉缩放比例
```

例如：

```text
50%：总览更多主线
100%：正常查看
150%：放大查看局部任务
200%：细看某个区域
```

## 3.9 主线过多时的显示控制

当主线过多时，需要：

```text
- 主线折叠
- 主线隐藏
- 主线分组
- 主线筛选
- 重要主线固定
```

## 3.10 主视图实现判别

```text
必须自定义插件实现。
```

建议模块：

```text
FishboneTimelineView

├─ TaskBranchRenderer
├─ RelationLineRenderer
├─ TaskCheckboxController
```

功能：

```text

- 日期星期
- 天气
```

实现：


# 5. 实现判别总表

| 层级 | 现成插件 | 自定义插件 | Codex Skill | 最终策略 |
|---|---|---|---|---|
| 0. 环境与执行层 | Local REST API / MCP 可选 | 必须 | 必须 | Codex 主控，自定义插件执行 |

| 4. 辅助模块层 | 优先复用 | 必要时自定义 | 每日总结 / 本周重点需要 | 现成插件优先，必要时魔改 |

# 7. Codex 工作约束

Codex 后续开发时必须遵守：

```text
1. 不要一次性开发完整系统。
2. 不要直接修改真实 Vault 中的旧笔记。
3. 所有真实 Vault 写入前必须有备份或 Git commit。
4. 所有测试先在 sample-vault 中完成。
5. 标准任务 md 是唯一数据源。
6. 现成插件只能辅助显示，不可反向定义核心数据结构。
7. 主视图必须自定义实现。
8. relations 字段是任务关系的统一入口。
9. mainline 数量、名称、颜色、排序必须用户自定义。
10. 每个阶段完成后必须输出完成内容、文件变更、测试结果、风险点和下一步建议。
```

# 8. 当前结论

当前 v2.4 架构已经确认：

```text
Codex：负责理解、生成、管理、写入、维护。
Skill：负责任务理解规则、每日总结规则、插件开发规则。
自定义 Obsidian 插件：负责标准任务层、鱼骨主视图、关键交互、快速输入、任务写入。
现成 Obsidian 插件：优先承担辅助模块，可通过 CSS / 配置 / 轻度 fork 魔改。
标准任务数据层：必须由我们自己定义，是所有视图和模块的唯一数据源。
```
