# M6.6-M6.9 详细计划

## 参考插件与借鉴边界

本阶段参考资料已下载到 `docs/reference-plugins/`。参考对象只用于学习交互和架构，不作为运行依赖，不改变当前个人规划系统的标准任务 schema。

- `quickadd-readme.md`：参考 Capture / Template / Macro / Multi choices 的输入流程设计。
- `obsidian-tasks-readme.md`：参考跨 Vault 聚合、查询、状态切换写回源文件。
- `dataview-readme.md`：参考把 Vault 当作可查询数据库的聚合思路。
- `projects-readme.md`：参考同一批笔记的 Table / Board / Calendar / Gallery 多视图组织方式，以及“尽量不污染笔记”的设计原则。
- `day-planner-readme.md`：参考日程时间线、每日笔记/Tasks/在线日历多来源聚合、当前时间块和时间追踪。
- `advanced-canvas-readme.md`：参考画布节点/边、元数据缓存、edge path、层级、命中区和可配置功能。
- `calendar-readme.md` 与 `periodic-notes-readme.md`：参考日期导航、每日/每周/每月笔记创建与打开逻辑。
- `etch-weather-readme.md`：参考天气信息以本地 Markdown 记录为主，而不是只显示临时 API 结果。

约束：

- 任务 md 仍是任务唯一可信数据源。
- `mainlines.json` 仍是主线唯一可信数据源。
- 外部插件不作为依赖，不读取它们的私有数据结构。
- 所有真实 Vault 写入必须由用户显式点击触发。
- M6.6-M6.9 仍聚焦右侧/下方工作台和输入能力，不重构 M5 鱼骨画布核心。

## M6.6：快速输入写入与任务候选确认

### 目标

把当前悬浮快速输入框从“仅候选预览”推进到可用的轻量任务创建入口。它和刚新增的“新建任务”按钮共用 `TaskRepository.createTask()`，但面向更快的自然语言输入。

### 参考方法

- QuickAdd：把输入拆成 Capture、Template、Macro 三层。我们不做复杂宏系统，但采用“输入 -> 候选 -> 确认写入”的流程。
- Obsidian Tasks：任何视图中完成或修改任务都必须写回源文件。
- Projects：创建时使用模板化字段，但不让插件状态污染业务笔记。

### 功能范围

1. 快速输入解析
   - 输入一句自然语言，生成一个 `QuickInputCandidate`。
   - 识别标题、日期、主线、优先级、状态。
   - 日期支持：
     - 今天、明天、后天。
     - `YYYY-MM-DD`。
     - 未识别时默认今天或 inbox，按设置决定。
   - 主线只允许匹配现有用户主线，不自动猜不存在的主线。

2. 候选确认 UI
   - 悬浮输入框下方显示候选卡片。
   - 支持直接确认写入。
   - 支持“展开编辑”，复用新建任务弹窗字段。
   - 支持取消候选。
   - 写入成功后刷新鱼骨画布、右侧模块、下方工作台。

3. 多任务拆分的边界
   - M6.6 先不做复杂多任务拆分。
   - 如果输入包含明显分隔符，如换行、分号，可提示“检测到多条，后续版本支持批量创建”。

4. 数据写入
   - 复用 `TaskRepository.createTask()`。
   - 写入标准 planning-task md。
   - `source_type: quick-input`。
   - `source_excerpt` 保存原始输入。
   - `review_status: confirmed`，因为由用户确认写入。

### 文件范围

- `plugin/src/views/FishboneTimelineView.ts`
- `plugin/src/data/taskRepository.ts`
- 可选新增 `plugin/src/intake/quickInputParser.ts`
- `scripts/validate-m6-6-quick-input.js`
- `tests/plugin/m6-6-manual-test-checklist.md`
- `DEVLOG.md`

### 验收标准

- 输入“明天 学习 看 Transformer 第4章 高优先级”后，候选中能识别日期、主线、标题和优先级。
- 点击确认后创建标准任务 md。
- 新任务立即出现在鱼骨画布、今日/本周模块、下方工作台对应列。
- 输入不存在的主线时不自动创建主线，候选显示为未分配或要求确认。
- 取消候选不写入任何文件。
- `npm run build` 和 M6.6 验证脚本通过。

### 不做

- 不接外部 LLM。
- 不做复杂批量任务拆分。
- 不做自动创建主线。
- 不做 relation 自动推断。

## M6.7：每日总结模块

### 目标

实现概念图中的“每日总结”模块，让系统可以从当前标准任务数据生成当天总结，并写入每日总结 md。

### 参考方法

- Periodic Notes / Calendar：按日期创建和打开每日笔记。
- Day Planner：把日程、任务和时间块按日期聚合。
- Obsidian Tasks / Dataview：从全 Vault 聚合任务状态。
- Etch Weather：重要上下文可以写入 Markdown，形成离线历史记录。

### 功能范围

1. 右侧每日总结模块
   - 显示今日总结是否已生成。
   - 显示最后生成时间。
   - 提供“生成总结 / 查看总结 / 重新生成”按钮。
   - 模块内部可滚动，不撑大整个右侧面板。

2. 总结数据聚合
   - 今日完成任务。
   - 今日新增任务。
   - 今日进行中/阻塞任务。
   - 过期未完成任务。
   - 主线进展变化。
   - 今日快速输入记录。
   - 明日建议。

3. 写入路径
   - 建议路径：`PlanningSystem/DailySummaries/YYYY/MM/YYYY-MM-DD_每日总结.md`。
   - frontmatter：
     - `type: daily-summary`
     - `date`
     - `generated_at`
     - `task_count`
     - `done_count`
     - `blocked_count`

4. 与 Skill 的关系
   - 后续可调用 `daily-summary-skill` 的规则生成更高质量文本。
   - M6.7 插件内先做 deterministic summary，不依赖外部模型。

### 文件范围

- 新增 `plugin/src/dashboard/dailySummary.ts`
- 新增 `plugin/src/data/dailySummaryRepository.ts`
- 修改 `FishboneTimelineView.ts` 增加模块渲染与按钮。
- 修改 `settings.ts` 增加模块顺序默认项。
- 新增 `scripts/validate-m6-7-daily-summary.js`
- 新增手动验收清单。

### 验收标准

- 当天存在任务时，右侧模块显示摘要状态。
- 点击生成后创建每日总结 md。
- 再次点击查看能打开对应总结文件。
- 重新生成只覆盖总结主体，不破坏 frontmatter。
- 无任务时也能生成“今日暂无任务”的总结。
- 构建和验证脚本通过。

### 不做

- 不做自动定时生成。
- 不接天气数据。
- 不做跨周/月复盘。
- 不调用外部 LLM。

## M6.8：时间 / 天气模块

### 目标

实现概念图中的“时间 / 天气”模块。优先做稳定、低风险的时间显示；天气采用可选、可缓存、可关闭的实现。

### 参考方法

- Etch Weather：天气结果可以写入 Markdown，形成离线历史，而不是每次都依赖 API。
- Calendar / Periodic Notes：默认从当前日期推导上下文。
- Day Planner：当前时间可以作为任务视图的实时上下文。

### 功能范围

1. 时间模块
   - 显示当前本地时间。
   - 显示日期和星期。
   - 每分钟刷新。
   - 可选显示当前日视图中心日期，避免和系统真实日期混淆。

2. 天气模块
   - 设置项中配置城市或经纬度。
   - 优先使用无 key 的 Open-Meteo。
   - 缓存当天天气结果到插件设置或 `PlanningSystem/WeatherCache/`。
   - 网络失败时显示上次缓存和失败提示。
   - 未配置位置时只显示时间，不显示错误。

3. 可写入每日总结
   - M6.8 先只提供“写入今日上下文”按钮。
   - 写入目标可以是每日总结 md 的天气段落。
   - 不自动写入，必须用户点击。

### 文件范围

- 新增 `plugin/src/dashboard/timeWeather.ts`
- 新增 `plugin/src/data/weatherRepository.ts`
- 修改 `settings.ts`：
  - `weatherLocationName`
  - `weatherLatitude`
  - `weatherLongitude`
  - `weatherUnit`
  - `enableWeather`
- 修改设置页增加天气配置。
- 修改右侧模块 registry，加入 `time-weather`。
- 新增 `scripts/validate-m6-8-time-weather.js`

### 验收标准

- 未配置天气时，模块正常显示时间，不报错。
- 配置经纬度后可拉取天气并缓存。
- 断网后显示缓存结果或明确提示。
- 时间每分钟更新，不触发整张鱼骨图重渲染。
- 关闭天气后不请求网络。
- 构建和验证脚本通过。

### 不做

- 不做多城市管理。
- 不做小时级复杂天气图表。
- 不默认写入每日笔记。
- 不强制要求用户申请 API key。

## M6.9：工作台模块管理与布局配置

### 目标

把 M6 工作台从“固定模块组合”推进到“可配置模块系统”，为 M7 以后继续扩展任务编辑、关系编辑、总结、天气模块打基础。

### 参考方法

- Projects：多视图组织和稳定优先原则。
- Dashboards / StartPage 类插件：模块化首页、模块顺序和布局配置。
- Advanced Canvas：功能可开关、默认设置可恢复、复杂交互要有清晰模式边界。

### 功能范围

1. 模块注册表
   - 定义统一模块元信息：
     - `id`
     - `title`
     - `area: right | bottom | floating`
     - `defaultHeight`
     - `minHeight`
     - `defaultVisible`
     - `render`
   - 右侧模块和下方模块统一由 registry 驱动。

2. 模块管理 UI
   - 工具栏新增“模块管理”按钮。
   - 弹窗中支持：
     - 显示/隐藏模块。
     - 恢复默认布局。
     - 重置模块高度。
     - 重置模块顺序。
   - 不做复杂拖拽设置页，拖拽仍在工作台内完成。

3. 持久化增强
   - `dashboardState.moduleOrder`
   - `dashboardState.moduleHeights`
   - `dashboardState.moduleVisibility`
   - `dashboardState.moduleCollapsed`
   - `dashboardState.workbenchColumnOrder`
   - `dashboardState.workbenchHeight`
   - 向后兼容旧设置，缺失字段自动补默认值。

4. 折叠状态
   - 每个右侧模块可折叠。
   - 折叠后只显示标题栏和计数。
   - 折叠状态持久化。

5. 下方模块收口
   - 下方状态工作台仍保留三列。
   - 为未来新增“筛选器 / 今日列表 / 快速入口”预留 bottom module 结构。
   - 但 M6.9 不新增新下方业务模块。

### 文件范围

- 新增 `plugin/src/dashboard/dashboardModules.ts`
- 修改 `settings.ts` 扩展 dashboardState。
- 修改 `FishboneTimelineView.ts` 使用模块 registry 渲染。
- 修改 `styles.css` 增加折叠和模块管理样式。
- 新增 `scripts/validate-m6-9-module-management.js`
- 新增手动验收清单。

### 验收标准

- 模块管理弹窗可隐藏/显示右侧模块。
- 模块顺序、显隐、高度、折叠状态刷新后保留。
- 恢复默认布局能回到内置顺序和高度。
- 旧 settings 中没有新字段时不报错。
- 工作台隐藏/显示按钮仍正常。
- 下方状态工作台不被模块管理破坏。
- 构建和 M6.1-M6.9 相关验证脚本通过。

### 不做

- 不做用户自定义模块脚本。
- 不做第三方模块市场。
- 不做复杂网格布局编辑器。
- 不引入外部 dashboard 插件依赖。

## 阶段顺序建议

1. 先做 M6.6，因为当前已经有 `TaskRepository.createTask()` 和“新建任务”按钮，快速输入写入可以最小成本闭环。
2. 再做 M6.7，每日总结依赖任务聚合和写入能力，适合在快速输入稳定后做。
3. 再做 M6.8，天气属于外部数据，应该在本地总结机制稳定后作为可选上下文接入。
4. 最后做 M6.9，把 M6 新增的模块统一收口，避免后面 M7/M8 扩展时继续堆硬编码。

## 统一验证矩阵

每个阶段完成后至少运行：

- `npm run typecheck`
- `npm run build`
- 当前阶段验证脚本。
- `scripts/validate-m6-5-workbench.js`
- 与任务写入相关时运行 `scripts/validate-m5-interaction.js`

真实 Vault 安装验收：

- 同步 `plugin/main.js`、`styles.css`、`manifest.json` 到真实 Vault 插件目录。
- 通过 Local REST API 执行 app reload。
- 打开鱼骨时间视图。
- 手动执行当前阶段清单。
